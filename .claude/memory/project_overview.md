Maelle is an AI executive assistant platform (**v1.8.4**) built in Node.js/TypeScript. **Mission: agent that works as a human EA.** Filter test for every decision: "would a real human EA do this?" — outranks speed, completeness, elegance. Runs primarily on Slack, backed by Microsoft Graph (Outlook calendar), Anthropic Claude Sonnet 4.6 for reasoning (Sonnet everywhere), SQLite via better-sqlite3. Per-user YAML profiles in `config/users/`. Multi-tenant: one deployment, N executives, one Slack app per assistant identity. **Runs under PM2 on the owner's laptop today** — two processes: `maelle` (the bot) and `maelle-deploy-watcher` (polls origin/master every 5 min, auto-pulls + rebuilds + restarts when a `Maelle Auto-Triage` commit lands; owner's own commits are skipped). PM2 ecosystem file at `ecosystem.config.js`. **v1.8.4 adds intent-routed outreach + forwarded huddle recaps + triage context restored:** `message_colleague` takes optional `intent` + `context` fields. When `intent='meeting_reschedule'` with `context={meeting_id, proposed_start, proposed_end}`, the outreach reply dispatcher routes the colleague's reply through `src/skills/meetingReschedule.ts` which calls `updateMeeting` to MOVE the existing event on approval (instead of coord creating a new one — the issue #26 bug shape). `outreach_jobs.intent` + `outreach_jobs.context_json` columns added (additive). `coordinate_meeting` now preflights against existing calendar events — if the requested subject substring-matches an existing event with overlapping participants, the tool refuses and points Sonnet to `message_colleague` with reschedule intent. **Four-layer model (don't violate):**

1. **Core (always on):** MemorySkill (`core/assistant.ts`), OutreachCoreSkill (`core/outreach.ts`), TasksSkill (`tasks/skill.ts`), RoutinesSkill (`tasks/crons.ts`). Plus engine infra: orchestrator, background loop, task runner + dispatchers + materializer + lateness, approvals resolver + orphan backfill, `core/assistantSelf.ts` (Maelle's own row in people_memory).

2. **Skills (togglable in profile YAML):** MeetingsSkill (`skills/meetings.ts` — direct ops + multi-party coord, merged from ex-scheduling + ex-coordination), CalendarHealthSkill, SearchSkill, ResearchSkill, SummarySkill (`skills/summary.ts` — v1.7.2, 3-stage transcript→summary→share state machine, profile YAML key `summary` since v1.7.6), KnowledgeBaseSkill (`skills/knowledge.ts` — v1.7.4, file-based markdown KB, profile YAML key `knowledge` since v1.7.6). CalendarHealthSkill profile YAML key is `calendar` since v1.7.6. Internal helper `skills/_meetingsOps.ts` (underscore = not a loadable skill; MeetingsSkill delegates to it for direct-op handlers). Registry at `skills/registry.ts`. Legacy YAML `scheduling`/`coordination` auto-migrate to `meetings`.

3. **Connections (comm-surface framework):** `connectors/slack/` (Bolt app, reply pipeline `postReply.ts`, coord state machine still domain-muddled — 1.7 target), `connectors/graph/` (calendar + free/busy), `connectors/whatsapp.ts` (stub). New `src/connections/slack/messaging.ts` (v1.7.2) is the FIRST piece of the planned Connection-interface migration (issue #1) — exposes `sendDM`/`sendMpim`/`postToChannel`/`findUserByName`/`findChannelByName`. SummarySkill is the first consumer; outreach.ts and coord.ts remain on legacy paths until issue #1 ports them. No formal `Connection` interface yet.

4. **Tools & Utilities:** `utils/claimChecker.ts` (honesty gate), `utils/dateVerifier.ts` (weekday/date retry), `utils/securityGate.ts`, `utils/coordGuard.ts`, `utils/imageGuard.ts` (v1.7.1 — Sonnet image-text injection scanner), `utils/rateLimit.ts`, `utils/shadowNotify.ts`, `utils/slackFormat.ts`, `utils/addresseeGate.ts`, `utils/genderDetect.ts`, `utils/logger.ts`. `src/voice/` (audio in/out) and `src/vision/` (v1.7.1 — image input) sit at the same layer as utilities — Slack input-format helpers, not domain skills. `db/` has per-table helpers. `config/` has profile loader + env.

**Core invariants to never break:**

- Task system is single source of truth for "what's on my plate" — `get_my_tasks` hydrates from linked tables (coord_jobs, outreach_jobs, approvals, pending_requests); LLM never fills from memory. `getOpenTasksForOwner` includes `completed` status (completed → informed two-step).
- Coord-terminal → approval sync lives inside `updateCoordJob` (`db/jobs.ts`): ANY transition to booked/cancelled/abandoned auto-resolves pending approvals AND cancels approval_expiry tasks. Single gate, impossible to forget.
- `message_colleague` (outreach.ts) and `initiateCoordination` (coord.ts) do NOT write to people_memory. Operational state lives in operational tables. `updateOutreachJob` and `updateCoordJob` write TERMINAL history to `people_memory.notes` only (past-tense: "Booked 'X' for <slot>", "Exchange: sent '...' → replied: '...'"). `app.ts` does NOT write colleague messages to notes.
- Find_available_slots enforces all schedule rules deterministically (work hours per day-type, lunch, thinking time per-office/home, quarter-hour alignment, NO buffer padding — durations bake it in). Meeting mode required: `in_person | online | either | custom` (+ travel_buffer_minutes for custom). Auto-expands up to 21 days.
- `delete_meeting` is idempotent per event_id per turn (orchestrator-level). Same id twice → ok:false reason='already_deleted_this_turn'.
- Empty orchestrator reply triggers a recovery pass (one grounded Claude call: describe what you did, ask clarifying Q, or NO_REPLY). No fabricated "Done."
- Claim-checker runs over every owner draft; false action claims (messaged X, booked Y) trigger a retry turn with `tool_choice` forcing the right tool.
- Date-verifier scans drafts for weekday/date pairs; mismatches vs the 14-day lookup trigger a corrective retry. Owner AND colleague paths.
- Assistant never says "the system / force the slot / threshold / policy" when talking to the owner — narrate rules as HIS preferences.

**Prompts vs code principle:** determinism → code (booking, delete, date alignment, slot rules, approval sync); judgment/tone → prompt. Don't cram determinism into prompts; don't cram judgment into regex.

**Prompt sizes (v1.7.0 baseline, after 1.6 pruning):** owner ~12k tokens (was 20k in 1.6.11, −39%), colleague ~9k (was 14.7k, −37%). Base prompt keeps only identity + dynamic data + auth + general honesty rules 1-8 + language + formatting + tone. Skill-specific rules live in each skill's `getSystemPromptSection`.
